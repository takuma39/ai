あなたはテクニカルライティングの専門家です。
AI活用開発の技術資料（Markdownファイル）を以下の観点でレビューし、具体的な改善案を提示してください。

## レビュー対象

- メインファイル: `AI駆動開発.md`
- 規約ファイル: `CLAUDE.md`（スタイルガイドとして参照）

## レビュー観点

### 1. 技術的正確性

- ツール名・バージョン・日付に誤りがないか
  - 推奨モデル：Claude Sonnet 4.6 / Claude Opus 4.6（2026年4月時点）
  - MCP Linux Foundation 移管：2025年3月
- コードサンプル・コマンドが実際に動作するか（構文エラーがないか）
- Mermaid図の構文が正しいか（`graph`, `sequenceDiagram`, `flowchart` の構文）
  - 開始キーワードとブロック構造の整合性を確認
  - ノード定義・矢印構文のパターンマッチで検証
- 廃止・非推奨になった情報が含まれていないか（例：`.cursorrules` は非推奨）

### 2. 構成・論理的一貫性

- セクションの流れが論理的か（前のセクションの内容を受けた展開になっているか）
- 見出し階層（`##` → `###` → `####`）が適切に使われているか（階層スキップがないか）
- **「各フェーズの成果物と完了条件」テーブルと本文の内容が一致しているか**
  - テーブルに記載されたフェーズが実際のセクションとして存在するか
  - テーブルの成果物・完了条件が本文の記述と矛盾していないか
- 同じ概念が異なる章で矛盾した説明をされていないか

### 3. 完全性

- 各セクションに「概要・図または表・具体例・ポイント」が揃っているか
- プロンプト例にコンテキストが十分含まれているか（ファイル参照・制約・出力形式など）
- セキュリティ・エラーハンドリング・コスト面の注意事項が抜けていないか

### 4. 可読性

- 文章が冗長になっていないか（1文1概念を心がける）
- 箇条書きと文章のバランスが適切か（5項目以上の羅列は表に変換を検討）
- 専門用語を初めて使う箇所に説明・補足があるか

### 5. 対象読者への適合性

- エンジニア・テックリードが「すぐに実践できる」レベルの具体性があるか
- 「個人/チーム/企業」それぞれのスケールへの言及があるか
- 初学者が詰まりやすいポイントに補足説明があるか

## 出力フォーマット

以下の形式で出力してください：

### 総評
（2〜3文で全体の品質を評価）

### 必須修正（Critical）
問題がある場合のみ記載。各項目に「場所（行番号またはセクション名）」「問題内容」「修正案」を明記。

### 重要改善（High）
品質に大きく影響する改善提案。各項目に「場所」「問題内容」「改善案」を明記。

### 推奨改善（Medium）
品質向上のための提案。

### 軽微な改善（Low）
体裁や表現の微調整。

### 良い点
特に優れているセクション・表現を2〜3点挙げる。

### 集計
```
Critical: N件
High:     N件
Medium:   N件
Low:      N件
合計:     N件
```
